# الخطة التطويرية الجديدة لمشروع ضاد — من الصفر إلى منتج عالمي

هذه الخطة تستبدل الخطة السابقة بالكامل. الفرق الجوهري أنها لا تعتبر “إنشاء ملف” مرحلة مكتملة، ولا تخلط البحث اللغوي بالواجهة والبنية التحتية. كل مرحلة لها مخرجات، قياسات، وموانع انتقال واضحة.

## تعريف النجاح

الوصول إلى “مستوى Grammarly للعربية” ليس ميزة واحدة، بل مجموع ستة محاور:

1. **الصحة اللغوية:** دقة مرتفعة وإنذارات كاذبة منخفضة على نصوص حقيقية.
2. **التغطية:** إملاء وصرف ونحو وأسلوب ولهجات وتشكيل واتساق مستند.
3. **السرعة:** استجابة فورية محليًا وعلى الويب.
4. **قابلية التفسير:** كل اقتراح يشرح السبب ومصدر القاعدة ودرجة الثقة.
5. **الانتشار:** محرر، API، إضافات، سطح مكتب، وأدوات مطورين.
6. **الثقة:** خصوصية واضحة، تصحيح آمن، إصدارات قابلة لإعادة البناء، وحوكمة لغوية.

## القواعد غير القابلة للتفاوض

- المستودع هو الحقيقة الوحيدة.
- كل ميزة لها Benchmark قبل وبعد.
- الدقة أهم من كثرة الاقتراحات.
- “غير متأكد” يعني الصمت أو عرض اقتراح منخفض الثقة، لا التصحيح التلقائي.
- الذكاء العصبي اختياري ويعزز النواة ولا يصبح مصدر الحقيقة الوحيد.
- بيانات المستخدم لا تدخل تدريبًا أو تجميعًا إلا بموافقة صريحة ومجهولة وآمنة.
- لا نكرر منطق التدقيق في كل عميل.
- العلامة التجارية والشعار بعد ثبات المنتج وهويته الوظيفية.

## Phase 0 — الحقيقة، السلامة، والهندسة الأساسية **✅ مكتملة**

### الهدف

تحويل النسخة الحالية من نموذج أولي جيد إلى خط أساس موثوق يمكن البناء فوقه.

### المخرجات

- تدقيق مستودع موثق.
- سياسة تصحيح تلقائي آمن.
- مخطط إعدادات واضح بدل الخيارات المبعثرة.
- تثبيت وبناء قابلان للتكرار.
- CI على Python المدعومة.
- فحص نوعي، تغطية، وأمن تبعيات.
- توحيد النسخة بين الكود والحزمة والـAPI.
- وصف صادق للميزات والخصوصية والتوافق.
- Seed benchmark مستقل عن أمثلة القواعد.

### بوابة الخروج

- جميع الاختبارات خضراء.
- بناء wheel وsdist ناجح في CI.
- لا اعتماد تشغيل ناقص.
- لا تطبيق تلقائي لاقتراح غير آمن.
- README لا يحتوي ادعاءً غير مثبت.
- تقرير خط أساس منشور: زمن، عدد القواعد، ونسبة FP/FN على مجموعة Seed.

## Phase 1 — بنية النص العربي وRule Engine v2 **✅ مكتملة في v0.3.0**

### الهدف

بناء طبقة نص صحيحة في Unicode ومحرك قواعد قادر على السياق والاستثناءات، قبل إضافة آلاف القواعد.

### المخرجات

- Tokenizer يحافظ على الإزاحات مع التشكيل، الرموز، الروابط، البريد، الأرقام، والكود المختلط.
- تقسيم جمل وفق العربية والاختصارات والقوائم.
- Normalization متعدد الأوضاع بدل دالة واحدة.
- Rule schema بإصدار رسمي وJSON Schema.
- أنواع قواعد: literal، regex، token sequence، context، exception، document.
- `confidence` و`autofix` و`tags` و`references` و`profiles`.
- حل تعارضات يعتمد الأولوية والثقة والنطاق، لا ترتيب الملفات فقط.
- Suppression محلي: تجاهل قاعدة/كلمة/سطر/مستند.

### بوابة الخروج

- اختبارات Unicode وoffsets تشمل 500 حالة على الأقل.
- صفر استبدال بإزاحة خاطئة في corpus الاختبار.
- زمن النواة الحتمية P95 أقل من 75ms لنص 500 حرف على جهاز مرجعي.
- كل قاعدة تمر Schema validation واختبارات مستقلة.

## Phase 2 — البيانات والتقييم قبل “الذكاء” **✅ مكتملة في v0.4.0**

### الهدف

إنشاء مسطرة قياس لا تتغير حسب الميزة التي نريد إثبات نجاحها.

### المخرجات

- Gold corpus متعدد المجالات: صحافة، أكاديمي، إداري، تعليمي، اجتماعي، ولهجات.
- فصل train/dev/test وعدم استخدام test في صناعة القواعد.
- إرشادات Annotation باللغة العربية.
- مراجعة مزدوجة لعينة وحساب اتفاق المراجعين.
- مقاييس span-level وcorrection-level وsentence-level.
- Precision، Recall، F0.5، FP/1000 words، MRR للاقتراحات.
- لوحة Regression لكل إصدار وفئة ولهجة ومجال.
- سجل تراخيص كل Dataset.

### بوابة الخروج

- 5,000 جملة ذهبية مضبوطة مبدئية على الأقل، مع تمييز الاصطناعي عن الطبيعي.
- 1,000 جملة بمرورَي مراجعة مستقلين، مع التصريح بنوع المراجعة.
- Benchmark قابل للتشغيل بأمر واحد.
- خط أساس منشور للإصدار الحالي بلا تجميل.

## Phase 3 — الإملاء المعجمي والتحليل الصرفي **✅ مكتملة في v0.5.0**

### الهدف

الانتقال من قائمة أخطاء محفوظة إلى فهم الكلمة وسوابقها ولواحقها ومرشحاتها.

### المخرجات

- واجهة `MorphologyBackend` مستقرة.
- backend خفيف محلي عالي الدقة.
- تكاملات اختيارية مع أدوات مرخصة بوضوح.
- قاموس ترددات ومفردات مجال وقاموس شخصي.
- Arabic-aware weighted edit distance.
- مرشح اقتراحات يجمع المسافة والتردد والسياق.
- دعم الأسماء والأعلام والمصطلحات وعدم اعتبارها أخطاء تلقائيًا.
- قواعد صرفية في Rule Engine v2.

### بوابة الخروج

- Precision إملائي ≥ 95% على test المستقل.
- Recall ≥ 85% كبداية مع خطة رفع معلنة.
- Top-3 suggestion accuracy ≥ 95% للأخطاء المعجمية المعروفة.
- P95 أقل من 100ms لنص 500 حرف محليًا.
- كل backend اختياري ويمكن تعطيله دون تعطيل النواة.

## Phase 4 — النحو والتركيب والإعراب المرشح **✅ النواة الحتمية مكتملة في v0.6.0**

### الهدف

بناء فحوص نحوية منضبطة، لا مجموعة Regex تحمل اسم “محلل نحوي”.

### المخرجات

- طبقة features صرفية موحدة.
- chunker للجملة الاسمية والفعلية والإضافة والنعت والجار والمجرور.
- فحوص المطابقة: الفعل والفاعل، النعت والمنعوت، العدد، الضمائر، الموصولات.
- إن وأخواتها وكان وأخواتها ضمن حدود الرؤية الممكنة للنص غير المشكل.
- Collocation للمتلازمات الفعلية وحروف الجر.
- Iʿrāb candidate API مع درجة ثقة وأسباب متعددة.
- قاعدة “الصمت عند الغموض”.

### حالة التنفيذ في v0.6.0

أُغلقت النواة الحتمية المطلوبة: سبعة أنواع علاقات وسبع عائلات تشخيصية عالية الدقة، مع API للإعراب المرشح وتحسن مقاس بلا زيادة في الإنذارات الكاذبة. يظل توسيع التغطية إلى 30 عائلة متنوعة هدفًا قبل الانتقال إلى Beta، لا ادعاءً منجزًا في هذه النسخة.

### بوابة الخروج الممتدة قبل Beta

- ≥ 30 عائلة فحص نحوي حقيقية، لا 30 مثالًا لنفس الفحص.
- False positives أقل من 5 لكل 1000 جملة على corpus نظيف.
- F0.5 يتجاوز خط أساس Phase 0 بفرق ذي دلالة.
- الإعراب لا يُعرض كحقيقة قطعية عندما تكون القراءة متعددة.

## Phase 5 — الوضوح، الأسلوب، والنبرة **✅ التنفيذ الهندسي مكتمل في v0.7.0**

### الهدف

تقديم قيمة كتابية تتجاوز “صح/خطأ” من دون فرض ذوق واحد على العربية.

### المخرجات

- مقاييس قابلية قراءة عربية موثقة.
- كشف الحشو، التكرار غير المقصود، المبني للمجهول، وطول الجمل.
- Profiles: أكاديمي، إداري، صحفي، تعليمي، ودّي، أدبي.
- اتساق النبرة والمخاطَب والمصطلحات.
- شرح الفرق بين الخطأ، التفضيل، والخيار الأسلوبي.
- لا Auto-fix للأسلوب.

### بوابة الخروج

> أُغلقت بوابات الكود والسلامة والانحدار في v0.7.0. تبقى البنود البشرية الآتية بوابة صلاحية خارجية قبل ادعاء جودة أسلوبية عامة.

- تقييم بشري مزدوج لعينة أسلوبية.
- ≥ 80% من الاقتراحات المصنفة “مفيدة” في الاختبار البشري.
- أقل من 10% اقتراحات تغيّر المعنى وفق المراجعين.

## Phase 6 — اللهجات وجسر اللهجة ↔ الفصحى **✅ التنفيذ الهندسي مكتمل في v0.8.0**

### الهدف

تحويل اللهجات من “قاموس استبدال كلمات” إلى تحليل سياقي يحترم هوية النص.

### المخرجات

- تحديد لهجة واحتمالات، لا افتراض لهجة واحدة.
- معاجم مصدّرة ومراجعة لكل منطقة.
- قواعد صرف ولهجاء للأحرف چ/گ/پ/ڤ/ژ.
- تحويل جملة وسياق لا كلمة منفردة فقط.
- الحفاظ على الأسماء والتعابير الثقافية.
- وضع شرح تعليمي للمقابل الفصيح.
- التحويل دائمًا opt-in.

### ما أُنجز في v0.8.0

- محرك تحديد احتمالي بخمس فئات: المصرية والشامية والخليجية والعراقية والمغاربية، مع فئة `mixed` عند تقارب الأدلة.
- تحويلات مفردة وعبارية وتركيبية، تتضمن تصريف فعل الإرادة وربط المضارع بـ«أن» وفق الشخص الظاهر.
- تحقق صرفي ونحوي لكل اقتراح تركيبي، وإزاحات أصلية ثابتة، وسياسة `autofix=false` إلزامية.
- أوضاع صريحة للتحويل: `dhad fix --dialects` و`Dhad.correct(mode="dialects")` و`dhad parse --msa`.
- عزل كامل لقياسات الإملاء والنحو والترقيم: نطاقها الميكانيكي لم يتغير مقارنة بـv0.7.0.

> أُغلقت بوابات الكود والسلامة والانحدار. تبقى مراجعة متحدثين أصليين على corpora طبيعية وحساب معدل تغيّر المعنى الجسيم بوابة صلاحية خارجية قبل ادعاء اكتمال Phase 6 لغويًا.

### بوابة الخروج

- Benchmarks منفصلة للعراقية والخليجية والشامية والمصرية أولًا.
- تقييم متحدثين أصليين لكل لهجة.
- أقل من 3% تغيّر معنى جسيم في عينة التحويل.

## Phase 7 — الطبقة العصبية الهجينة **✅ التنفيذ الهندسي مكتمل في v0.9.0**

### الهدف

استخدام النموذج فقط حيث يضيف قيمة قابلة للقياس، مع فصل صارم عن النتائج الحتمية.

### المخرجات

- `NeuralBackend` محلي وسحابي اختياري.
- Trust gateway مع provenance لكل اقتراح.
- سياسات خصوصية ومفاتيح المستخدم وحدود البيانات.
- إعادة صياغة opt-in.
- تقييم كل نموذج مقابل نفس Benchmark.
- اقتراح قواعد من الأنماط المتكررة، لا كتابة قواعد “مقبولة” تلقائيًا.
- تنقية PII وتجميع k-anonymity قبل أي تعلم مجتمعي.

### ما أُنجز في v0.9.0

- بوابة ثقة تعيد ترتيب القراءات الصرفية الموجودة فقط، ولا تتدخل في التحليلات عالية الثقة.
- نموذج n-gram خطي محلي ومفسر، وواجهة HuggingFace حقيقية ذات تحميل كسول.
- فئة `neural_suggestion` منفصلة و`autofix=false`، مع أولوية مطلقة للنتائج الحتمية.
- ثبات Benchmark الرسمي وFP/1000 دون أي زيادة.

### بوابة الخروج

- الطبقة الحتمية تعمل كاملة عند تعطيل العصبي.
- لا اقتراح عصبي يطبق تلقائيًا.
- تحسن F0.5 أو تقييم المنفعة البشرية بوضوح مقابل التكلفة والزمن.
- نموذج محلي واحد موثق على الأقل.

## Phase 8 — التشكيل، الدلالة، واتساق المستند **✅ التنفيذ الهندسي مكتمل في v0.10.0**

### الهدف

دعم القراءة الدلالية والعمل على المستند الكامل بعد نضج الصرف والنحو.

### المخرجات

- تشكيل فك الالتباس، تشكيل أواخر، وتشكيل كامل.
- N-best readings مع confidence.
- كشف الأعلام والمصطلحات والاختصارات.
- glossary للمشروع.
- اتساق أسماء الأشخاص والمنظمات والأرقام والتواريخ.
- كشف تناقض الزمن السردي والضمائر عند الثقة الكافية.

### ما أُنجز في v0.10.0

- محرك تشكيل بثلاثة أوضاع يفصل التشكيل الداخلي عن حركة الإعراب، ويحافظ على النص والإزاحات الأصلية.
- الاستفادة من القراءة الصرفية المعاد ترتيبها سياقيًا ومن `IrabCandidate` للنصب والجزم والجر والرفع.
- مورد دلالي متحقق منه بـJSON Schema لتوحيد الصيغ، وأنماط الأرقام، وقواعد التكرار الدلالي المحافظة.
- `DocumentConsistencyTracker` للمستند الكامل أو الأجزاء المتدفقة، مع اختيار أول صيغة موثوقة مرجعًا للاتساق.
- كشف تناقض صريح شديد التحفظ بمقارنة الفعل المثبت والمنفي، مع حواجز للسياق الزمني.
- فئات مستقلة `diacritics` و`semantics` و`consistency`، وكلها `autofix=false`.
- ثبات Benchmark الرسمي والنطاق الميكانيكي دون أي زيادة في FP/1000.

### بوابة الخروج العلمية المتبقية

- قياس WER/DER للتشكيل على Dataset طبيعي مستقل لم يُنجز بعد.
- قياس دقة اتساق المصطلحات على corpus وثائقي بشري لم يُنجز بعد.
- قياس مستند 50 صفحة على عتاد مرجعي موثق لم يُنشر بعد.

## Phase 9 — API وSDK ونواة التكامل **✅ النطاق المطلوب مكتمل في v0.11.0**

### الهدف

إنشاء عقد منتج مستقر قبل بناء خمسة عملاء مختلفين.

### ما أُنجز في v0.11.0

- REST API غير متزامن بإصدار `/api/v1` ونماذج Pydantic صارمة وOpenAPI/Swagger/ReDoc.
- مسارات `check`, `parse`, `diacritize`, `style`, و`dialect` مع الحفاظ على LanguageTool v2.
- SDK بايثون موحد للتشغيل المحلي أو عبر HTTP، مع واجهات sync/async ونماذج استجابة واحدة.
- خادم LSP 3.17 مستقل عبر stdio: diagnostics، hover صرفي/نحوي، وQuick Fix الآمن.
- مزامنة LSP كاملة ومتزايدة، وإزاحات UTF-16 صحيحة، واختبارات JSON-RPC framing.
- Contract/compatibility tests وSemVer للإصدار العام.

### ما بقي من النطاق الموسع

- SDK TypeScript.
- REST document sessions وrequest cancellation.
- SARIF وJSONL للـCLI.
- سياسة deprecation منشورة وآلية compatibility matrix أوسع.

### بوابة الخروج المنجزة للنطاق المطلوب

- عقود API وLSP مختبرة آليًا.
- توافق الخادم القديم محفوظ.
- Benchmark النواة مطابق لـv0.10.0 بلا زيادة في FP.

## Phase 10 — تجربة المنتج والمنصات **✅ النطاق الهندسي المطلوب مكتمل في v0.12.0**

### الهدف

نشر النواة بعد ثبات العقد، لا قبل ذلك، مع إبقاء كل عميل طبقة رقيقة فوق API Phase 9.

### ما أُنجز في v0.12.0

- واجهة ويب RTL متجاوبة منفصلة إلى HTML/CSS/JavaScript، تستدعي `check` و`style` و`dialect` و`diacritize` وتعرض مبررات الاقتراحات وسياسة الأمان.
- PWA قابلة للتثبيت مع manifest وأيقونات maskable وService Worker يخزن غلاف الواجهة فقط ولا يخزن نص المستخدم أو ردود التحليل.
- تشغيل FastAPI بواجهة كاملة أو API فقط عبر `create_app(serve_web=False)` و`dhad serve --no-web`.
- سياسة CORS افتراضية local-only، ودعم origins لإضافات المتصفح، ورفض wildcard، ورؤوس CSP وPermissions-Policy وX-Frame-Options.
- إضافة Chrome/Edge Manifest V3 كاملة: Service Worker مركزي، إعداد خادم محلي/بعيد بصلاحية اختيارية، Content Script لـ`textarea` و`contenteditable`، خطوط متموجة، وبطاقة اقتراحات.
- ربط أزرار التطبيق بحقل `autofix`: التصحيح الآمن مميز، والأسلوب واللهجة يظهران كمراجعة صريحة.
- مشغّل سطح مكتب `dhad-desktop`/`dhad desktop` يدير خادم loopback ويستخدم pywebview أو وضع التطبيق في Chromium/Edge أو المتصفح الافتراضي.
- أساس هاتف صريح عبر PWA؛ لا ادعاء بلوحة مفاتيح أصلية في هذه النسخة.
- 17 اختبارًا جديدًا لعقود الويب وPWA وManifest V3 وJavaScript والمشغل، مع ثبات Benchmark.

### بوابات لم تُغلق بعد

- نشر Chrome Web Store وEdge Add-ons، ثم Firefox/Safari بحزم واختبارات مستقلة.
- E2E مرئية على مصفوفة متصفحات حقيقية؛ بيئة البناء الحالية منعت Chromium من التنقل الشبكي بسياسة إدارية، لذلك لم يُدّع اختبار بصري غير منفذ.
