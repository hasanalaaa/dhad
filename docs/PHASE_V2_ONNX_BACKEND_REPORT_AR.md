# تقرير: طبقة الاستدلال العصبي عبر ONNX Runtime (`OnnxBackend`)

**المشروع:** ضاد (Dhad) — المساعد الكتابي العربي مفتوح المصدر
**النطاق:** إضافة خلفية استدلال عصبية حقيقية إلى الطبقة الهجينة، دون المساس بالمسار الحتمي ولا بالسلوك الافتراضي.
**التاريخ:** 2026-07-22 · **الأساس:** `1.0.0-rc1` + خارطة V2

---

## 1. الدافع والقرار المعماري

الطبقة العصبية القائمة صُمِّمت حول عقد واحد نظيف هو بروتوكول `NeuralBackend` (`backend.py`):

```python
class NeuralBackend(Protocol):
    @property
    def name(self) -> str: ...
    @property
    def available(self) -> bool: ...
    def score(self, request: NeuralRequest) -> tuple[CandidateScore, ...]: ...
```

يوجد بالفعل تنفيذان لهذا العقد: `StatisticalContextBackend` (نموذج n-gram خطي متفرّق، افتراضي وقابل للتدقيق) و`TransformerBackend` (يحمّل نموذج HuggingFace اختياريًا عبر torch). الإضافة هي **الشقيق الثالث** لهذا العقد: `OnnxBackend`، وهو نظير `TransformerBackend` لكن على زمن تشغيل محايد وخفيف — `onnxruntime` + محلّل HuggingFace السريع — بلا اعتماد على torch/transformers.

القرار الجوهري: **لم نُعِد كتابة الطبقة الهجينة ولا غيّرنا الافتراضيات.** المحرك الافتراضي يبقى النموذج الإحصائي. `OnnxBackend` مسار تفعيل صريح (opt‑in) للنشرات التي تملك نموذج فك التباس مُصدَّرًا إلى ONNX. هذا يحافظ على ضمانات المشروع الأساسية: بوابة الثقة لا تتجاوز التحليل الحتمي عالي الثقة، ولا تُولِّد مرشحات من خارج قراءات الصرف، وكل اقتراح احتمالي يبقى `autofix=false`.

## 2. عقد التنفيذ (تكافؤ مع `TransformerBackend`)

`OnnxBackend.score()` يتبع نفس خط أنابيب `TransformerBackend` بدقّة:

1. **تعليم الهدف**: يُعاد بناء نص الجملة مع تغليف الكلمة الهدف بعلامتَي `[TARGET] … [/TARGET]` (نفس `_marked_text`).
2. **الترميز**: عبر `tokenizers.Tokenizer.from_file("tokenizer.json")` (المسار الأساسي، بلا torch)، مع سقوط اختياري إلى `transformers.AutoTokenizer` عند توفّره فقط.
3. **الاستدلال**: تمريرة أمامية حقيقية عبر `onnxruntime.InferenceSession`.
4. **الإسقاط على المرشحات**: تُطابَق `id2label` مع تسميات مرشحات الطلب، وتُستبعد أي تسمية للنموذج ليست ضمن مرشحات الطلب.
5. **إعادة التطبيع**: softmax مستقر عدديًا، ثم تطبيع الاحتمالات الناجية لتجمع إلى 1، وترتيبها تنازليًا.

### ربط المدخلات المقود بالرسم البياني (I/O binding)

بدلًا من افتراض أسماء مدخلات ثابتة، يبني `OnnxBackend` قاموس التغذية من مدخلات الرسم البياني المُعلَنة فعلًا (`session.get_inputs()`)، ويحوّل كل مصفوفة إلى **نوع التنسور الذي يتوقّعه ذلك المدخل بالضبط**. هكذا تعمل النماذج المُصدَّرة بـ`input_ids` من نوع int64 و`attention_mask` من نوع int32 معًا دون أي إعداد. المصفوفات تُمرَّر مباشرة كـ`numpy` (المسار الأكفأ لـonnxruntime).

## 3. الصدق في `available` (بلا اختلاق)

`available` لا تُحمّل النموذج ولا ترفع استثناءً؛ ترجع `True` فقط حين تتوفر **كل** المكوّنات في البيئة الحالية:

- وجود حزمة `onnxruntime`،
- وجود ملف الأوزان `model.onnx` على القرص،
- إمكانية حلّ محلّل (ملف `tokenizer.json` + `tokenizers`، أو دليل + `transformers`)،
- إمكانية حلّ خريطة تسميات (قائمة `labels` صريحة أو `config.json` يحمل `id2label`).

عند غياب أي مكوّن، تُبلّغ الخلفية عن عدم توفّرها، فتتحوّل الطبقة الهجينة **تلقائيًا وبشفافية** إلى المسار الحتمي: `HybridNeuralEngine.refine_sentence` تعود مبكرًا، و`report` تتخطّى الاقتراحات. لا نتيجة مُختلقة أبدًا.

## 4. التحميل الكسول

لا يُستورد `onnxruntime` ولا `tokenizers` ولا `numpy` إلا عند أول استدلال. إنشاء الكائن لا يلمس القرص إلا لحلّ المسارات (رخيص)، ولا يبني `InferenceSession` قبل أول `score`. التحميل محميّ بـ`threading.Lock` ليكون آمنًا تحت الخدمة غير المتزامنة/متعدّدة العمّال.

## 5. مِعمار النشر ومساعد البيئة

التخطيط المتوقَّع على القرص يطابق تصدير `optimum` القياسي:

```
model_dir/
    model.onnx        # رسم التصنيف المُصدَّر
    tokenizer.json    # محلّل HuggingFace سريع
    config.json       # يحمل id2label (ما لم تُمرَّر labels صراحةً)
```

التفعيل صريح:

```python
from dhad import HybridNeuralEngine, OnnxBackend, onnx_backend_from_env

engine = HybridNeuralEngine(backend=OnnxBackend("/models/dhad-wsd-onnx"))
# أو عبر متغيّر بيئة، دون تغيير الافتراضي:
backend = onnx_backend_from_env("DHAD_NEURAL_ONNX_MODEL")  # يعيد None إذا لم يُضبط
```

## 6. الاختبارات (استدلال حقيقي، بلا محاكاة)

أُضيف `tests/test_onnx_backend.py` (10 اختبارات)، وهو يبني في زمن التشغيل **رسم تصنيف ONNX حقيقيًا صغيرًا** (عبر `onnx.helper`) و**محلّل HuggingFace سريعًا حقيقيًا**، ثم يمرّرهما عبر `onnxruntime` من طرف إلى طرف. لا شيء مُموَّه: أوزان حقيقية، ترميز حقيقي، تمريرة أمامية حقيقية. أبرز ما تُثبته:

- تحقّق البروتوكول (`isinstance(be, NeuralBackend)`) وكسل التحميل (`_session is None` قبل `score`).
- استدلال حقيقي: الاحتمالات في `[0,1]`، مجموعها ≈ 1، مرتّبة تنازليًا، ومُسقَطة على مرشحات الطلب فقط.
- **مدخلات مختلطة الأنواع** (int64 لـids و int32 للقناع) لإثبات التحويل الصحيح لكل مدخل.
- التسميات الصريحة تعمل بلا `config.json`.
- `available=False` عند غياب الأوزان أو المحلّل، وصحّة تحقّق الإعدادات (نص فارغ، `max_length<16`، `labels` فارغة).
- **بوابة صادقة داخل المحرك**: خلفية غير متوفّرة تُبقي `HybridNeuralEngine` عملية فارغة والقراءة الحتمية سليمة؛ وخلفية متوفّرة تُوصَل فعليًا (توجيه الكلمة الملتبسة عبر مُقيّم ONNX).
- الاختبار يُتخطّى بلطف (`importorskip`) عند غياب حزم ONNX.

## 7. التحزيم

- مجموعة اعتماد اختيارية جديدة: `onnx = ["onnxruntime>=1.17", "tokenizers>=0.19"]`.
- أُضيفت الحزم إلى `dev` كي يشغّل التكامل المستمر اختبارات ONNX فعليًا بدل تخطّيها.
- تصدير `OnnxBackend` و`onnx_backend_from_env` من `dhad.neural` ومن الحزمة العليا `dhad`.

## 8. التحقّق

- **`ruff check .` نظيف** على كامل المشروع (سطر ≤ 100).
- **مجموعة الاختبارات كاملة: 1755 نجاحًا، 1 تخطّيًا** (اختبار تكافؤ Rust يتطلّب امتداد pyo3 المبنيّ — غير متعلّق بهذا التغيير). الفارق مقابل الأساس هو بالضبط اختبارات ONNX العشرة الجديدة، بلا أي انحدار.
- زمن التشغيل: onnxruntime 1.27، tokenizers 0.23، مزوّد CPU.

## 9. حدود صادقة والخطوة التالية

- الخلفية **بنية تحتية**: تُضيف قيمة لغوية فعلية فقط عند تزويدها بنموذج فك التباس عربي مُصدَّر إلى ONNX؛ النموذج الصغير في الاختبارات لإثبات المسار لا للجودة اللغوية.
- النطاق الحالي هو **فك الالتباس الدلالي (WSD)** المقيّد بمرشحات الصرف — كما `TransformerBackend` تمامًا — وليس مجموعات الالتباس الإملائي السياقي (تلك تبقى من اختصاص النموذج الإحصائي عبر `candidate_labels`).
- الافتراضي هو مزوّد CPU (اختيار صحيح وقابل لإعادة الإنتاج لخدمة تدقيق)؛ المزوّدات قابلة للتهيئة.
- التالي: تصدير/تقطير نموذج عربي حقيقي (مثل BERT عربي مُقطَّر) إلى ONNX، ونشر مقاييس p50/p99 للاستدلال، وربما توسعة CLI بعلم `--onnx-model`.
